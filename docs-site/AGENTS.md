# docs-site

Aether kernel's public documentation site: an Astro + Starlight site that
syncs content from the main repo's `docs/` tree (plus a few other README/ADR
sources) and deploys to Cloudflare Workers. `CLAUDE.md` in this directory is
a symlink to this file.

## Change content scope here, not in code

`content.manifest.txt` is the one file to edit to add/remove what the site
publishes — one glob per line, relative to the repo root. `npm run sync`
(`scripts/sync-content.mjs`) reads it and copies matching files into
`src/content/docs/`.

## The trap: most of src/content/docs/ is generated

Every file under `src/content/docs/` is generated by `npm run sync` **except**
`index.md` and `agent-quickstart.md`, which are hand-authored and belong to
this site specifically. `npm run sync` deletes and regenerates everything
else on every run — hand-editing a synced file is silently thrown away the
next time sync runs.

- To change generated content: edit the source in the main repo's `docs/`
  tree (or wherever `content.manifest.txt` points), then re-run `npm run
  sync`.
- To change site-only content (landing page, agent quickstart): edit
  `src/content/docs/index.md` or `src/content/docs/agent-quickstart.md`
  directly.

`sync-content.mjs` also rewrites relative links in synced Markdown
(`rewriteRelativeLinks`) so cross-links between synced docs keep working once
the files move under `src/content/docs/`.

## Build pipeline order

1. **`npm run sync`** — copies manifest-matched files into
   `src/content/docs/`, rewrites relative links.
2. **`astro build`** — renders pages and runs the `starlight-links-validator`
   plugin, which **fails the build on any broken internal link**. This is
   the authoritative link check for this site — if `astro build` passes,
   internal links are valid.
3. **`node scripts/emit-markdown-pages.mjs`** — writes a raw `.md` twin into
   `dist/` for every rendered page, so each page is servable as either HTML
   or plain Markdown.

`npm run build` runs all three steps in order.

## Two test suites, split by runtime

- `npm test` — plain Node (Vitest default config, `vitest.config.js`):
  covers `scripts/sync-content.mjs` and `scripts/emit-markdown-pages.mjs`.
- `npm run test:worker` — the Cloudflare Worker (`vitest.workers.config.js`,
  `@cloudflare/vitest-pool-workers`): covers `worker/entry.js`. Split out
  because it needs the real `workerd` runtime, which the default Node test
  environment can't provide. Uses a small fixture set under
  `worker/test-fixtures/` instead of the real `dist/` build output, so these
  tests don't depend on site content.

Run both before considering a change done.

## What the Worker does

`worker/entry.js` implements content negotiation for AI agents: a request
whose path ends in `.md`, or whose `Accept` header includes
`text/markdown`, gets served the page's raw Markdown twin (emitted by
`emit-markdown-pages.mjs`) instead of the rendered HTML. Everything else
falls through to normal static-asset serving via the `ASSETS` binding.

## Local dev / build / preview

```bash
npm run dev      # sync + astro dev (localhost:4321)
npm run build    # sync + astro build + emit-markdown-pages -> dist/
npm run preview  # wrangler dev — preview the built Worker locally
npm run check    # sync + astro check (types)
npm test         # plain-Node test suite
npm run test:worker  # Cloudflare Worker test suite (needs workerd)
```

Requires Node >= 22.13.0 (see `engines` in `package.json`).
