# Home Assistant 集成

Home Assistant 由 AetherEdge 在家庭局域网内连接。AetherCloud 不直接连接用户的
Home Assistant，不保存访问令牌，也不成为设备实时状态的权威来源。

目标数据链路如下：

```text
Home Assistant
  → AetherEdge 连接器
  → 与厂商无关的集成拓扑和观测
  → 受治理的 CloudLink 传输
  → AetherCloud 投影
  → 智能体发现、解释和规划
```

这条只读链路已经可以在显式启用的源码构建中运行。AetherEdge 使用两个可跨崩溃恢复的
CloudLink 流发布严格的 alpha.4 候选版拓扑与观测；AetherCloud 验证当前会话和运行时清单
声明后，以原子方式保存业务投影、重放回执、审计证据、发件箱证据和连续应用确认。

它仍是实验性集成，不是已经发布的家庭产品。公共协议尚未正式发布，部分云端组合仍使用
内存适配器；可选的智能体查询适配器也还没有公开网络服务。生产级注册和凭据生命周期
尚未完成。

## 权责和密钥边界

| 事项                                     | 权威来源                                          |
| ---------------------------------------- | ------------------------------------------------- |
| Home Assistant 注册表和实体观测状态      | Home Assistant 提供证据，由 AetherEdge 在本地消费 |
| 归一化、重连、重新同步和本地接纳         | AetherEdge                                        |
| 实时点位、确定性规则、安全联锁和物理执行 | AetherEdge                                        |
| 公共拓扑和观测结构                       | AetherContracts                                   |
| 可搜索的历史投影和智能体上下文           | AetherCloud                                       |

Home Assistant 地址、访问令牌、刷新令牌和授权客户端材料只能存在于边缘本地配置或密钥
服务中。它们不得进入拓扑、观测、CloudLink 消息、云端存储、审计详情、日志、提示词
或智能体上下文。

## 公共结构候选版本

AetherContracts 当前相邻工作区中的 `0.1.0-alpha.4` 候选版本已定义闭合且与厂商无关
的集成结构：

- 带无损代次的完整拓扑快照；
- 区域、设备、实体，以及一个实体下的多个类型化点位；
- 由拓扑代次约束的观测批次；
- 精确保留的有符号整数、无符号整数、小数、有限浮点数、字符串、布尔值和字节；
- 明确的正常、不确定、错误和不可用质量；
- 对重复身份、悬空引用、类型不匹配及质量和值冲突的显式拒绝。

Home Assistant 注册表条目标识作为稳定实体标识；可变的 `entity_id` 只作为来源地址
保留，因此重命名不会产生一个新的 Aether 实体。`light`、`switch` 和 `fan` 的主开关
状态统一为布尔点 `is_on`；其他经过选择且有大小限制的属性会成为独立点位，任意属性不会
原样复制到公共结构。

公共结构明确排除凭据、任意服务调用和 CloudLink 传输封装。设备控制必须使用后续独立
定义、默认拒绝并带执行回执的能力结构。

AetherCloud 现有的完整消费者锁仍固定在已经发布的 `0.1.0-alpha.3`。只有 alpha.4
正式发布、按摘要导入并通过消费者测试后，云端才能宣称通过 alpha.4 一致性验证。当前
云端领域模型与候选字段保持一致，但不能替代这道发布门槛。

## 已实现的只读链路

三个仓库目前已经具备：

- AetherEdge 中经过认证的 Home Assistant WebSocket 适配器；
- 完整快照、有序状态变化、可跨重启恢复的拓扑代次、缺口检测和完整重同步；
- 严格的 alpha.4 候选版集成与 CloudLink 编解码；
- 拓扑和观测各自独立的边缘文件队列；
- MQTT 重连和精确重放；只有严格的当前会话应用确认才能删除队列记录；
- 新 CloudLink 会话中的运行时清单声明恢复和再次认证；
- 与厂商无关的云端拓扑和观测领域类型；
- 内部命令的闭合输入校验，以及对存储返回结果的完整复验；
- 由网关凭据限定范围的拓扑上报和观测上报命令；
- 由租户和项目限定范围的按标识投影查询与有界目录查询；
- 代次隔离、类型校验、幂等重放和冲突处理；
- 内存和 PostgreSQL 两种投影适配器；
- 在同一个 PostgreSQL 事务中绑定集成事实、收件箱身份、不可变回执、拓扑历史、审计记录、
  发件箱记录、CloudLink 传输身份和连续应用确认；
- 跨会话、跨凭据代次的精确重放，且不会重复业务副作用；
- 对提供方未来时间戳的有界时钟偏差校验；
- 明确标记为“边缘上报副本、不是实时权威状态”的查询结果。

目录按网关标识和集成标识稳定排序，只返回有大小限制的摘要字段。智能体可以先发现已有
投影副本，再读取一个精确详情。这个目录不是实时设备清单；云端接收时间只提供新鲜度
证据，不能证明设备当前物理状态。只有显式注入对应应用查询后，可选的智能体资源适配器
才会出现。

内存适配器仍只用于开发。PostgreSQL 集成适配器已经通过真实数据库的重启和并发测试，但
当前 CloudLink 会话、运行时清单和部署组合仍包含内存实现，因此还不能称为完整生产云进程。

## 重新同步

AetherEdge 会先订阅变化，再获取初始快照，避免发现过程中发生的状态变化被静默丢失。
注册表变化、实体删除、连接中断、非法协议数据或观测序列缺口都会触发完整重新同步。
AetherCloud 同样要求观测批次绑定准确的拓扑代次。任何一端都不会猜测缺失状态、把旧值
带入未知拓扑，也不会把“未知”或“不可用”伪造成普通字符串值。

## 当前用户边界

- 正式发布并按摘要固定的 alpha.4 完整消费者导入；
- AetherEdge 预编译包、安装器配置和正式的 Home Assistant 兼容性基线；
- 由安装器管理的注册流程、消息代理权限模板、凭据轮换和生产密钥适配器；
- CloudLink 会话与运行时清单全部使用 PostgreSQL 的生产组合；
- 面向生产的接口和智能体发现入口；
- 有界历史观测查询和完整的新鲜度模型；
- 楼层、标签、配置来源和服务能力的公共发布；
- 可在生产环境启用的受治理控制；独立的
  `aether.cloudlink.integration-control.v1alpha1` 电源动作仍是默认关闭的实验能力，
  并且不能宣称物理动作已经完成；
- 面向生产的授权接入和令牌轮换。

配置和持久发布步骤见
[AetherEdge 的 Home Assistant 指南](https://docs.aetheriot.workers.dev/guides/home-assistant/)。
以上缺口仍是正式产品门槛。仓库测试通过不能被描述成已经支持生产部署。
