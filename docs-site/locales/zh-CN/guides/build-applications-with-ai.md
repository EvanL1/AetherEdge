# 使用 AI 构建应用

AetherEdge 在设计上是无头的。它提供了人工智能代理构建特定于站点的应用程序所需的确定性边缘运行时和机器可读合约。生成的 Web UI、移动应用、CLI 或后端是可替换的客户端：它永远不是实时状态、所需状态或安全机构的一部分。

本指南定义了参考应用程序和下游产品共享的开发方法。

## 可选代理技能

可选择将仓库的 `aether-iot` 技能添加到兼容的编码助手。 Bun仅用于运行第三方`skills`CLI；此命令不会安装 AetherEdge 运行时或 Rust SDK：
```bash
bunx skills add EvanL1/AetherEdge --skill aether-iot
```

技能故意很小。它教授工作流程并将代理路由到当前的在线文档，而不是嵌入过时的副本。文档服务发布：

- `/llms.txt` - 精选文档索引；
- `/llms-full.txt` - 完整的公共语料库；
- 通过附加 `.md` 或发送 `Accept: text/markdown` 为每个页面提供一个 Markdown 双胞胎。

当代理还需要实时运行时信息时连接 `aether mcp`。默认的 MCP 表面是只读的；不要仅仅为了生成应用程序而启用写入。

## 使用合约堆栈

应用程序应按以下顺序发现 AetherEdge：

| 源 | 它建立的内容 |
|---|---|
| 运行时清单 | 确切的内核版本、目标、特性、协议和功能 |
| 活动Pack | 领域词汇、支持的资产和兼容性要求 |
| 开放API | 精确的 HTTP 路径、架构、状态代码、身份验证和命令策略 |
| MCP | 可供 AI 客户端使用的实时工具和结构化运行时结果 |
| 在线Markdown | 架构、工作流程、安全指南和领域解释 |

请勿用 README 示例或模型内存替换正在运行的版本的 OpenAPI 文档。在版本契约明确之前，请勿假设已安装的 Pack、可选适配器或写入功能已存在。

## 从只读用户故事开始

将第一个提示写入有界结果，例如：
```text
Build a read-only operations page for this AetherEdge site. Show service health,
the current topology revision, point values with quality and freshness, active
alarms, and the last hour of available history. Do not add device controls.
```

在生成代码之前，代理应生成简短的功能清单：

1. 运行时和 Pack 版本；
2. 页面所需的查询和订阅；
3. 满足它们的公共功能；
4. 缺失或仍属于内部的功能；
5. 用于客户端生成的确切 OpenAPI 文档。

缺失的公共查询是应用程序边界差距。不允许公开内部服务端口、抓取 SQLite、附加到 SHM 或创建端点。

## 保留一个远程边界

仅 `aether-api` 用于远程应用程序流量。它在 `/api/v1/io`、`/api/v1/automation`、`/api/v1/history`、`/api/v1/uplink` 和 `/api/v1/alarm` 下发布固定的经过身份验证的命名空间；它们所属的进程 API 仍处于环回状态。

因此，生成的远程客户端必须：

- 仅通过端口 6005 上经过身份验证的 `aether-api` 调用这些网关前缀功能；
- 使用正在运行的网关的 OpenAPI 合约来生成类型和请求；
- 使用其经过身份验证的 WebSocket 合约进行实时更新可用时；
- 报告不可用的用例，而不是代理内部端口；
- 使凭据远离源代码控制、URL、日志和浏览器持久性，而不是为机密而设计。

本地开发工具可以检查边缘主机上的环回 OpenAPI 文档，但这不会将这些端口转换为受支持的远程接口。

## 保留 IoT 状态语义

有用的 IoT 屏幕需要的不仅仅是数值。保留并显示以下内容的契约字段：

- 点标识和工程单元；
- 价值加上运行状况或质量；
- 源时间戳和观察到的新鲜度；
- 物理拓扑纪元；
- 所需的配置修订；
- 活动、降级或协调待处理运行时状态。

绝不静默将丢失、过时、不健康或代不匹配的数据强制为零。切勿将所需状态和活动运行时状态合并为一种乐观状态。当拓扑更改正在进行时，保持最后一个连贯生成可见或显示连贯视图不可用。

演示文稿名称、组、单元、枚举和推荐的可视化属于活动域包或下游应用程序。通用客户端不得从 AetherEdge 内核推断能量语义。

## 将命令添加为单独的阶段

仅在只读工作流程完成后引入命令。对于每个命令，呈现并执行服务器声明的合约：

- 风险和所需权限；
- 显式确认策略；
- 幂等性和预期修订行为；
- 请求关联和审核结果；
- 已接受、降级和不确定结果。

保留返回的 `request_id`， `command_id`，以及由此产生的修订。在超时、不完整的审核、已接受但降级的响应或未知的物理结果后，不要自动重试非幂等命令。设备命令接受证明本地命令平面接受了请求；需要反馈遥测来显示物理执行。

后端强制执行这些规则。确认对话框是有用的演示，但它绝不是安全边界。

## 验证生成的应用程序

每个维护的引用或下游应用程序都应通过：

1. 类型检查、linting、单元测试和生产构建；
2. 针对健康、陈旧、未经授权、降级和不确定结果的合约测试夹具；
3. 检查是否没有任何请求针对直接进程端口、SHM 或 SQLite；
4. 控制组件和写入凭据的只读默认值不存在；
5. 针对安全空或模拟 AetherEdge 组合进行集成检查。

参考应用程序演示契约并可以替换。它们不定义 API 行为、配置权限或域真相。
